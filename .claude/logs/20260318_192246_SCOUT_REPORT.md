## Suspected Root Cause Areas

1. **Comment accuracy (lib/agent_monitor.sh:86-87)**: The comment states stderr is teed "to both a dedicated file and the FIFO (for existing monitoring)" but the actual redirect `> "$_fifo" 2> >(tee -a "$_stderr_file" >&1)` sends stderr only to `$_stderr_file` and the caller's original stdout, NOT the FIFO. The process substitution `>(...)` expands before the main `> "$_fifo"` redirect is applied to the claude process. Stderr never reaches the FIFO. This is a documentation accuracy issue only; the actual implementation (dedicated file capture) works correctly and classification succeeds.

2. **api_auth retry exception (lib/errors.sh & future M13 retry logic)**: The `api_auth` subcategory is correctly marked transient per M12.1 spec (line 268: "All upstream errors are transient"). However, authentication failures won't self-resolve on retry—the user's API key won't magically become valid by waiting. The non-blocking note flags this as needing special handling: when M13 adds a retry envelope, it must explicitly check for `api_auth` and skip auto-retry, surfacing the error immediately instead of backing off. Currently there is no such skip logic, leaving a gap between error classification (M12.1) and retry implementation (M13).

3. **Unsafe echo in _match_pattern() (lib/errors.sh:37)**: The function uses `echo "$text"` which is unsafe on some implementations. Inputs beginning with `-n` can be misinterpreted as flags, and inputs containing escape sequences (`\t`, `\\`, etc.) may be expanded by the shell instead of passed literally. The fix already exists in `redact_sensitive()` in the same codebase, which uses `printf '%s\n'` for portability. This inconsistency creates a potential source of silent failures in error classification.
