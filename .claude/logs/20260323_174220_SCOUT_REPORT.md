## Key Symbols

### Core Functions (lib/health.sh)
- `assess_project_health(project_dir)` — Main assessment entry point, returns composite score 0-100
- `reassess_project_health(project_dir)` — Re-assessment with delta computation, calls assessment functions and loads previous baseline
- `_calculate_composite_score(test_score, quality_score, deps_score, docs_score, hygiene_score)` — Weighted composite calculation
- `_map_score_to_belt(score)` — Maps 0-100 score to belt label (White→Black)

### Dimension Check Functions (lib/health_checks.sh)
- `_check_test_health(project_dir)` — Returns score 0-100 based on test files, framework detection, naming conventions
- `_check_code_quality(project_dir)` — Returns score 0-100 based on linter config, pre-commit hooks, magic number density, TODO/FIXME density, function length heuristics, type safety
- `_check_dependency_health(project_dir)` — Returns score 0-100 based on lock file presence, commitment, bloat ratio, vuln scanner config, version freshness
- `_check_doc_quality(project_dir)` — Delegates to existing `assess_doc_quality()` from `detect_doc_quality.sh` or computes independently
- `_check_project_hygiene(project_dir)` — Returns score 0-100 based on `.gitignore`, `.env` committed, CI/CD detection, README instructions, CHANGELOG presence

### Supporting Functions
- `_sample_source_files(project_dir, count)` — Deterministic file sampling (sorted, reproducible)
- `_count_magic_numbers(file_list)` — Numeric literal density analysis
- `_count_code_comments(file_list)` — TODO/FIXME/HACK/XXX frequency
- `_parse_lock_file_versions(lock_file)` — Extract version strings from package-lock.json, Pipfile.lock, go.sum, etc.
- `_check_git_tracked_file(filepath)` — Safe git ls-files check for `.env` security

## Suspected Root Cause Areas

### Weight Validation
- `lib/config.sh` line ~50 — Must validate that sum of `HEALTH_WEIGHT_*` equals 100. If weights don't sum to 100, composite score will be incorrectly scaled

### Magic Number Detection Noise
- `lib/health_checks.sh` — Magic number detection inherently has false positives. Must skip common constants (0, 1, -1, 2, 100, 1000, 255, 256, 360, 1024, etc.) and focus on suspicious contexts (inside conditionals, function arguments). Err toward fewer false positives

### Test Health Without Execution
- `lib/health_checks.sh` — Without `HEALTH_RUN_TESTS=true`, test health is inferred from file count ratio. Report must clearly state: "Estimated from file presence. Run with HEALTH_RUN_TESTS=true for actual pass rate."

### Dependency Version Parsing
- `lib/health_checks.sh` — Lock file version strings are not always semver (e.g., git commit hashes, custom version schemes). Must parse gracefully and skip non-semver versions. "2+ majors behind" heuristic requires clean semver parsing

### File Sampling Determinism
- `lib/health_checks.sh` — Must use `git ls-files | sort` pattern (not random). Same repo state must produce identical score across runs for test stability

### Belt System Acceptance
- `lib/health.sh` — Some users may find the belt system patronizing. Ensure `HEALTH_SHOW_BELT=true` is the default for fun, but the 0-100 number is always visible. Configurable via `HEALTH_BELT_LABELS` for custom labels

### .env Security Check
- `lib/health_checks.sh` — Never read `.env` file contents (may contain secrets). Only check if FILENAME is tracked: `git ls-files .env`. This is a security-critical detail

### Dashboard Integration
- `lib/dashboard.sh` — `emit_dashboard_health()` must handle the case where `HEALTH_BASELINE.json` doesn't exist (first run after --init). Emit zeros or "N/A" for delta, not errors

### Intake Prompt Context
- `stages/intake.sh` — Building `HEALTH_SCORE_SUMMARY` must parse `HEALTH_REPORT.md` and extract low-scoring dimensions. Focus on actionable insights (not raw scores) for the PM agent
