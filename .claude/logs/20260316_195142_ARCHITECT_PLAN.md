## Out of Scope

### wc -l portability concern

**Observation claim:** `stages/cleanup.sh` and `stages/coder.sh`/`stages/tester.sh`
all use the `wc -l | tr -d '[:space:]'` idiom, creating a coordinated-update risk.

**Verification findings:** The exact `wc -l | tr -d '[:space:]'` pattern appears in:
- `stages/cleanup.sh:59` — `echo "$batch" | wc -l | tr -d '[:space:]'`
- `lib/agent.sh:228` — `wc -l < "$_summary_path" 2>/dev/null | tr -d '[:space:]'`
- `lib/agent.sh:325` — `wc -l < "$expected_file" | tr -d '[:space:]'`

`stages/coder.sh` and `stages/tester.sh` do NOT use this pattern — they use bare
`tr -d '[:space:]'` for string whitespace stripping, which is unrelated.

**Justification for out of scope:** The pattern is portable, correct, and spans only
2 files (3 call sites). The observation's "coordinated update" risk is real but low
— `wc -l` is POSIX-standard and the `tr` stripping addresses BSD/GNU output
differences reliably. Extracting a `_count_lines()` helper for 3 call sites would
add abstraction without proportional benefit. The observation premise (spanning
coder.sh and tester.sh) does not match the actual code. Not scheduled for a
pipeline run.
