## Non-Blocking Notes
- `app.js` is 751 lines — the 300-line ceiling in `reviewer.md` targets shell libraries; this is the first frontend JS file in the project. Functionally correct. Log for potential future split if it grows further.
- `_copy_static_files()` docstring says "Only overwrites if source is newer than destination" but the implementation unconditionally copies with `cp` (no `-nt` check). Code behavior is fine (always-overwrite is correct); comment is misleading. Fix the comment.
- `trendArrow()` in `app.js` assumes `runs[0]` is the most recent run (`recent = runs.slice(0,10)`). If `_parse_run_summaries` returns oldest-first (chronological order), trend arrows would be inverted. Assumption is not documented. Validate against actual parser output ordering.
- `emit_dashboard_run_state()` constructs `waiting_for` JSON via `printf %s` with empty-string expansion then fixes malformed JSON with bash string substitution (`${json/\"waiting_for\":,/\"waiting_for\":null,}`). Works correctly but fragile — a simpler approach would be to set the field to `null` or `"value"` using an explicit conditional before the printf call.

## Coverage Gaps
- `_copy_static_files()` is only asserted to exist in `dashboard.sh` (test 14); no test verifies that calling `init_dashboard()` actually copies `index.html`/`style.css`/`app.js` into the destination directory.
- `_regenerate_timeline_js()` verbosity filter paths (`minimal`, `normal`, `verbose`) are untested — no assertion that different verbosity levels produce different output sizes.
- `emit_dashboard_run_state()` `refresh_interval_ms` injection is asserted present in the shell source (test 15) but there is no integration test that calls the function and inspects the generated JS for the correct value.

## Drift Observations
- `lib/dashboard.sh:62-76` — `_copy_static_files()` documentation/implementation mismatch (always-overwrite vs. only-if-newer). Low severity but sets a misleading expectation for future maintainers.
- `app.js:656-669` — `trendArrow()` hardcodes `runs.length < 20` as the minimum for trend comparison and uses `slice(0,10)` / `slice(10,20)`. The data order assumption (newest-first) is load-bearing but not enforced or tested anywhere in the pipeline.
