## Naming Normalization

**1. Inconsistent `tr -d` argument across line-counting idiom in planning stage files**

The drift log flagged the `wc -l | tr -d` line-counting idiom as a coordination
risk. Codebase scan confirms the pattern exists in 4 planning-phase stage files
with an internal inconsistency: `stages/cleanup.sh` uses `tr -d '[:space:]'`
(POSIX character class, strips all whitespace) while the other three files use
`tr -d ' '` (literal space only). On macOS, `wc -l` can prepend leading spaces
to its output; `tr -d ' '` handles that. On Linux, no leading spaces appear.
Both work today, but the inconsistency adds confusion and `tr -d '[:space:]'` is
the safer, idiomatic form.

**Affected locations:**
- `stages/cleanup.sh:59` — uses `tr -d '[:space:]'` ← canonical form
- `stages/plan_generate.sh:60` — uses `tr -d ' '` ← normalize
- `stages/plan_generate.sh:89` — uses `tr -d ' '` ← normalize
- `stages/plan_interview.sh:355` — uses `tr -d ' '` ← normalize
- `stages/plan_followup_interview.sh:129` — uses `tr -d ' '` ← normalize
- `stages/plan_followup_interview.sh:180` — uses `tr -d ' '` ← normalize

**Action for jr coder:** In the five locations listed above (not cleanup.sh),
change `tr -d ' '` to `tr -d '[:space:]'`. No logic changes. Run
`bash -n` and `shellcheck` on all modified files after.

Note: The drift log originally cited `stages/coder.sh` and `stages/tester.sh`
as containing this pattern — codebase scan found no matches in those files.
The observation is valid but the file list was imprecise. The correction above
reflects actual locations.
