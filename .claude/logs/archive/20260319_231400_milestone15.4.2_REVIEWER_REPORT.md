## Non-Blocking Notes
- `lib/notes.sh` is 535 lines — nearly double the 300-line ceiling. The pre-existing NON_BLOCKING section (Milestone 5, ~143 lines) already pushed it over 300 before 15.4.1 added the seven new single-note functions (~192 lines). The spec explicitly places these functions in `lib/notes.sh`, so this milestone is correct as implemented, but a follow-on cleanup milestone should extract the single-note utilities (lines 199–391) into `lib/notes_single.sh` and source it from `tekhton.sh`.
- `_escape_sed_pattern()` (line 208) does not include `]` in its escape set. The sed pattern `s/[.[\*^$()+?{|/&]/\\&/g` will miss `]` characters. The function is marked unused (retained for 15.4.2/15.4.3), but should escape `]` before it's put into service. The same gap exists in the pre-existing `mark_note_resolved`/`mark_note_deferred` patterns at lines 503 and 527 — those pre-date this milestone.
- `claim_human_notes()` (lines 73, 78) and `resolve_human_notes()` (lines 138, 144, 156, 179, 190, 194) still use `sed -i` rather than the tmpfile+mv pattern the new functions use. This pre-existing inconsistency should be harmonized in a cleanup pass.

## Coverage Gaps

## Drift Observations
- `lib/notes.sh` now mixes three distinct concerns: (1) bulk human-notes operations for the legacy `--with-notes` flow, (2) single-note operations for `--human` mode, and (3) NON_BLOCKING_LOG.md operations for the Milestone 5 cleanup sweep. The file split into `notes_single.sh` noted above would also address this semantic mixing.
- `count_human_notes()` (line 36) uses `[ ]` instead of `[[ ]]` — inconsistent with the new functions which all use `[[ ]]`. Pre-existing, but worth harmonizing in the cleanup pass.
