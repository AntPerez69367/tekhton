## Non-Blocking Notes
- `stages/coder.sh:326-336`: Context budget logging captures `HUMAN_NOTES_BLOCK` BEFORE the notes-gating clear at line 345. When notes are not claimed, the context report will log a non-zero size for "Human Notes" even though the rendered prompt gets an empty block. Functionally correct (the prompt is right); the metrics are cosmetically misleading. Fix: move `_add_context_component "Human Notes" "$HUMAN_NOTES_BLOCK"` to after the should_claim_notes gate at line 346.
- `lib/notes.sh:20`: `should_claim_notes` uses `echo "$task_text" | grep -qiE` but the pattern already has `[Hh]` and `[Nn]` character classes, making `-i` redundant. More idiomatic: use a here-string `grep -qiE '...' <<< "$task_text"` to avoid the subshell and match project conventions in other files.
- `lib/config_defaults.sh:127`: `USAGE_THRESHOLD_PCT=0` was added as groundwork for a future feature (usage threshold pausing) but no acceptance criterion covers it and it appears in no other code path. Fine as a seed, but worth documenting in a comment that this is a stub for a not-yet-implemented feature to avoid confusion for future readers.
- Behavioral note — `should_claim_notes` + `NOTES_FILTER` interaction: Previously, `--notes-filter BUG "fix login crash"` would claim BUG notes because `HUMAN_NOTE_COUNT > 0`. With the new gating, `should_claim_notes "fix login crash"` returns 1 (false), so notes are silently skipped unless `--with-notes` is also passed. Users who relied on `--notes-filter TYPE` as the sole signal to address notes will now need `--with-notes --notes-filter TYPE`. This is consistent with the milestone spec but is an undocumented breaking change in workflow behavior. Consider adding `NOTES_FILTER != ""` as a third condition in `should_claim_notes` to preserve the old intent, or document the required migration in the help text.
