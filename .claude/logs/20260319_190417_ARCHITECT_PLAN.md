## Staleness Fixes

**Item 1 — Add explanatory comment to `resolve_human_notes()` regarding `_PIPELINE_EXIT_CODE`**

- **File**: `lib/notes.sh`, line ~182
- **Issue**: The `elif [[ -n "${_PIPELINE_EXIT_CODE:-}" ]] && [[ "${_PIPELINE_EXIT_CODE}" -eq 0 ]]`
  guard is correct but opaque. A reader unfamiliar with tekhton.sh's finalization sequence will
  not know that `_PIPELINE_EXIT_CODE` is always set by the caller immediately before this function
  is invoked (see `tekhton.sh:991`), making the `elif` always true in production. The guard exists
  solely for testability — callers in tests can set it to 1 to simulate failure. Without a comment,
  future maintainers risk misreading it as a safety guard or removing it as dead code.
- **Action**: Add a one-line comment above the `elif` explaining: (a) `_PIPELINE_EXIT_CODE` is
  set to 0 by `tekhton.sh` before calling this function when the pipeline succeeds, (b) the
  `elif` path fires only when `CODER_SUMMARY.md` is absent but pipeline exit was clean, and
  (c) the variable exists to allow test harnesses to simulate failure by setting it to non-zero.
- **Route**: Jr coder

## Simplification

**Item 3 — Consolidate dual `_check_gitignore_safety` + `git add -A` calls in `tekhton.sh`**

- **Files**: `tekhton.sh:1028–1029` and `tekhton.sh:1133–1136`
- **Issue**: The safety check and staging operation is performed twice in the same pipeline
  finalization path. First at line 1028 (pre-commit message generation block) and again inside
  `_do_git_commit()` at line 1135. This splits staging responsibility across two locations:
  - The outer block calls `_check_gitignore_safety` + `git add -A` to stage changes for
    `generate_commit_message()` (which reads `git diff --cached --stat`)
  - `_do_git_commit()` calls `_check_gitignore_safety` + `git add -A` again before committing
  The result is that `_check_gitignore_safety` fires twice per pipeline run, `git add -A` runs
  twice, and the staging + safety-check contract is distributed across two separate code locations
  with different responsibilities.
- **Action**: This is the correct candidate for the M15.3 `finalize_run()` consolidation. As
  part of that refactor, the staging responsibility should be owned by exactly one place. The
  recommended approach: move the initial `git add -A` (for commit message generation) into
  `generate_commit_message()` itself or into `_do_git_commit()` with a `--dry-run` diff read
  beforehand. `_check_gitignore_safety` should then be called exactly once, inside `_do_git_commit()`,
  immediately before the `git commit`. Remove the outer standalone `_check_gitignore_safety` +
  `git add -A` block (lines 1028–1029). Verify that `generate_commit_message()` still has access
  to accurate staged-change data after the consolidation.
- **Route**: Sr coder (architectural: changes the responsibility boundary of `_do_git_commit()`)
- **Note**: The drift log explicitly flags this for the M15.3 `finalize_run()` consolidation.
  This remediation should be scoped to or coordinated with that milestone, not implemented in
  isolation — it shares the same code region being refactored.
