## Structured Project Index (V3 post-release milestones)

**One intent. Many hands.**

V3.71.0 adds **5 milestones** across two themes: a complete rewrite of the project
index subsystem from a monolithic markdown file into a structured data layer, and
prompt engineering improvements that push recurring reviewer findings down to zero
before they ever reach review.

## Highlights

### Structured Project Index (M67–M69)

The project index has been completely rearchitected. Previously, `PROJECT_INDEX.md`
tried to serve three incompatible roles simultaneously — human-readable map,
programmatic index, and LLM prompt context — and failed at all three by truncating
data at a hard character budget the moment it was created.

- **Structured data layer (M67)** — `crawl_project()` now emits individual files
  to `.claude/index/`: `meta.json` (scan metadata), `tree.txt` (complete directory
  tree, no `head -500` truncation), `inventory.jsonl` (one record per file,
  unbounded), `dependencies.json`, `configs.json`, `tests.json`, and
  `samples/<file>.txt` (individual content samples with a manifest). All writes
  are atomic (mktemp + mv). `_list_tracked_files` is called exactly once per
  crawl (was 4 times). `PROJECT_INDEX_BUDGET` config key replaces hardcoded
  `120000` at all call sites.

- **Consumer migration to structured index (M68)** — New `lib/index_reader.sh`
  provides a reader API (`read_index_summary`, `read_index_meta`,
  `read_index_inventory`, `read_index_dependencies`, `read_index_configs`,
  `read_index_tests`, `read_index_samples`) that delivers bounded, structured
  content to each caller. Three pre-existing consumer bugs are fixed:
  - **Intake** was silently receiving empty project context for any project with
    more than ~100 files (`_safe_read_file` rejects files over 8KB — it was never
    a truncating reader). Now uses `read_index_summary` with an 8KB budget.
  - **Synthesis** was compressing the index via `summarize_headings` which kept
    only heading lines, destroying file inventory tables and all sampled content.
    Now uses `read_index_summary` with a 60KB budget.
  - **Replan** was injecting the full raw markdown (up to 120KB+) into the replan
    prompt with no budget gate. Now uses `read_index_summary` with a 40KB budget.
  - `_extract_sampled_files` regex bug fixed (was matching `^### \`` with a
    backtick that the crawler never emitted — sampled file change detection has
    been silently broken since M18).
  - All reader functions fall back to legacy `PROJECT_INDEX.md` parsing for
    projects that haven't re-crawled since M67.

- **Markdown view generator, rescan rewrite & legacy cleanup (M69)** — New
  `lib/index_view.sh` provides `generate_project_index_view()` which assembles a
  bounded `PROJECT_INDEX.md` from structured data using record selection (not
  string truncation). The `... (truncated to fit budget)` marker is gone. The
  rescan no longer does surgical markdown patching via `_replace_section`
  (eliminating the ARG_MAX risk on macOS for large sections); it updates affected
  structured files and regenerates the view. `_truncate_section` and
  `_replace_section` are deleted. The synthesis compression cascade that was
  applying `summarize_headings` to index content is removed. Legacy projects
  auto-migrate on the first rescan or crawl.

### Code Quality & Prompt Engineering (M70–M71)

Analysis of 14 historical `REVIEWER_REPORT.md` files showed ~60% of non-blocking
reviewer findings were issues the coder could have caught itself. File length
violations alone accounted for ~38% of all non-blockers, appearing on the same
files repeatedly.

- **Coder pre-completion self-check (M70)** — `prompts/coder.prompt.md` has a
  mandatory Step 5 self-check inserted into the execution order (before the
  coder sets COMPLETE). It covers: file length verification (`wc -l` on every
  touched file, hard 300-line ceiling), stale reference detection after renames,
  dead code removal, and consistency checks (new files in CODER_SUMMARY.md and
  architecture docs). `templates/coder.md` strengthens the 300-line rule and
  adds pipeline-failure consequence language to the write-first CODER_SUMMARY.md
  requirement. A new `## Observed Issues (out of scope)` section gives the coder
  a structured channel to record problems it noticed but didn't fix, instead of
  silently ignoring them or creating scope-creep non-blockers.

- **Tekhton shell hygiene rules (M71)** — `.claude/agents/coder.md` (Tekhton's
  project-level coder role) gains a `### Shell Hygiene` section with concrete
  rules targeting the most common bash non-blockers: `grep || true` under
  `set -e`, SC2155 two-line `local` pattern, `--` option terminator before
  variable arguments, sourced-file `set -euo pipefail` prohibition, stale
  reference grep after renames, and the 300-line file ceiling. Each rule
  includes a pattern or example. No changes to the reusable `templates/coder.md`
  or any pipeline infrastructure — this is project-scoped configuration.

## Get started

```bash
git clone https://github.com/geoffgodwin/tekhton.git
cd tekhton && chmod +x tekhton.sh
cd /path/to/your/project
/path/to/tekhton/tekhton.sh --init
```

`MILESTONE_ARCHIVE.md` for individual milestone records.
